# 验证阶梯 —— 验证强度由风险等级决定，不由心情决定

## 定级

先问三个问题，任何一个「是」就往上升一级：

1. 碰不碰钱（计费、支付、退款、额度）？→ 🔴
2. 碰不碰生产状态（迁移、切流、权限、配置）？→ 🔴
3. 改不改业务行为（逻辑、接口、开关、调度）？→ 至少 🟡

## 阶梯

### 🟢 低风险：文案、样式、纯展示

- [ ] 静态检查（lint）通过
- [ ] 类型检查 / 构建通过
- [ ] 改动处目标冒烟：亲眼看到改动生效的样子

### 🟡 中风险：业务逻辑、接口、开关

🟢 全部，加上：

- [ ] 单测 / 集成测试覆盖改动路径
- [ ] **隔离运行实例中实际触发目标路径**——不是「测试过了」，是「这条路真的被走过了」
- [ ] 功能开 / 关两态都验证（关闭态 = 原始行为，不是报错）

### 🔴 高风险：钱、权限、迁移、生产

🟡 全部，加上逐项证明：

- [ ] **金额**：正反向都对（多收、少收、不收三种情况的用例）
- [ ] **幂等**：同一操作重放两次，结果不变
- [ ] **权限**：无权限方真的进不来
- [ ] **迁移**：新旧数据双版本兼容，已应用的迁移不修改（改已应用迁移 = 发布必挂）
- [ ] **补偿**：失败后怎么补、补偿本身是否幂等
- [ ] **回退**：回退方案先于上线方案存在，且演练过一次
- [ ] 等价环境（staging/candidate）真实流量形态验收

## 三条铁律

1. **编译或单测通过 ≠ 目标场景验收。** 绿灯只证明照亮的地方没问题，不证明照对了地方。
2. **不信任何 agent 自报的绿灯。** 验收在主线程发生或由主线程复跑；证据（命令 + 关键输出）进账本。
3. **说不出怎么回退，就还没准备好上线。**
